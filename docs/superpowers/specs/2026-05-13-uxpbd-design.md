# UXPBD — Unified eXtended Position-Based Dynamics Solver

**Status:** Design (v1 spec, CSLC seams reserved for v2)
**Date:** 2026-05-13
**Author:** Nataliya Nechyporenko (drafted with Claude)
**Branch:** `xpbd_test`
**Target location:** `newton/_src/solvers/uxpbd/`

---

## 1 Motivation

Newton currently exposes three independent position-based solvers, each strong in one dimension and weak in others:

| Solver | Strengths | Limitations |
|---|---|---|
| `SolverXPBD` | Articulated rigid bodies with `body_q`/`body_qd`, full joint constraints (PRISMATIC, REVOLUTE, BALL, FIXED, FREE, DISTANCE, D6), motors, limits, restitution, `body_parent_f` reporting; soft FEM tetrahedra; springs; bending | Contact momentum approximated via `1/N` con-weighting (paper-flagged); no fluids; no shape-matched rigid; no compliant contact surface |
| `SolverSRXPBD` (PBD-R, Abderezaei et al.) | Tiled shape-matched rigid groups with **momentum-conservation constraint** + modified velocity update (`enforce_momentum_conservation_tiled`); inter-group particle-particle contact; solver-agnostic analytical-benchmark suite | No articulated rigid bodies, no joints, no soft FEM, no fluids |
| UPPFRTA (Macklin 2014, not in Newton) | Unified particle representation: rigid (via shape matching), soft, cloth, **liquid/fluid**, gas; phase-based collision filtering; position-level Coulomb friction; mass-scaling shock propagation | No articulated rigid with joints; not differentiable end-to-end at contact; contact averaging not strictly momentum-preserving |

UXPBD unifies all three into one solver. The unification thesis: **one neighbor-finding pass, one constraint loop, one friction model, one collision matrix — handling articulated robots interacting with rigid manipulanda, soft deformables, and liquids, with momentum-correct contact forces on the rigid side.**

This is needed for robot manipulation research in the `cslc_xpbd/` line of work: pick-and-place, compliant grasping of deformables, pouring — scenarios that today require switching solvers per scene type.

## 2 Goals and non-goals

### 2.1 Goals (v1)

1. Ship `SolverUXPBD` as a public Newton API at `newton.solvers.SolverUXPBD`.
2. Handle four substrates in one solver: articulated rigid (with kinematic lattice shells), free shape-matched rigid (momentum-correct), soft (springs / bending / FEM tet), liquid (PBF density constraint).
3. Pass all of Newton's existing solver test suite for articulated-rigid scenarios.
4. Demonstrate three scenarios end-to-end with regression-tested example scripts: pick-and-place (rigid), compliant grasp (deformable), pouring (liquid).
5. Quantitative MuJoCo / Hydroelastic comparison for the **rigid-only** subset (box push, ball drop / bounce).
6. Preserve every architectural seam needed to drop in **CSLC v2** (compliant sphere lattice contact) without data-structure migration.
7. Differentiability: `requires_grad` code paths preserved (no crashes); validation deferred to v2.

### 2.2 Non-goals (v1)

- CSLC quasistatic contact solve (deferred to v2; seams reserved per §5).
- Soft body **self-collision** enabled by default (off in v1; cloth self-collision via particle pipeline still works).
- Multi-phase fluids / immiscible liquids (deferred to v3+).
- Gas / smoke (UPPFRTA §7.2; no scenario demand).
- Gradient correctness validation (deferred to v2).
- Equality / mimic joint constraints (same scope as `SolverXPBD`).
- Hierarchical multi-resolution particles.
- Runtime particle addition (`add_fluid_particles` after `finalize()`); deferred to v3.

## 3 Architecture

### 3.1 Substrate model

Every collidable entity is represented as one or more particles in the **unified** `state.particle_q` / `state.particle_qd` arrays. The substrate is determined by per-particle metadata.

| Substrate | Authoritative state | Internal dynamics constraint | Generated by |
|---|---|---|---|
| 0 — Articulated rigid (per link) | `body_q ∈ SE(3)`, `body_qd ∈ se(3)` per link. Lattice particles are *kinematically driven* by `body_q`. | None (lattice particles inherit from body); joints constrain `body_q`. | `builder.add_lattice(link, ...)` or `add_lattice_to_all_links(...)` |
| 1 — Free shape-matched rigid | Particle `x_i`, `v_i`. Rigid pose implicit (recovered via SRXPBD shape match). | `solve_shape_matching_batch_tiled` + `enforce_momentum_conservation_tiled` | Existing `Box.add_morphit_spheres` / `add_spheres` pattern |
| 2 — Soft body | Particle `x_i`, `v_i` + tet / spring / edge connectivity | `solve_springs`, `bending_constraint`, `solve_tetrahedra` | Existing `add_soft_grid`, `add_soft_mesh`, `add_cloth_grid`, `add_cloth_mesh` |
| 3 — Fluid | Particle `x_i`, `v_i` + density-constraint phase | `solve_fluid_density_constraints` (PBF) | New `add_fluid_grid(...)` / `add_fluid_particles(...)` |

Per-particle metadata fields added to `Model`:

```python
model.particle_phase:     wp.array[wp.int32]   # UPPFRTA phase ID (collision gating)
model.particle_substrate: wp.array[wp.uint8]   # 0=lattice, 1=SM-rigid, 2=soft, 3=fluid
model.particle_group:     wp.array[wp.int32]   # substrate-dependent group index
```

**Phase ≠ substrate.** Phase decides who-collides-with-whom (UPPFRTA-style; same phase = same body = no contact). Substrate decides what kernel governs internal dynamics. The split lets a soft body and a shape-matched rigid coexist with the same phase if the user wants them to not collide.

### 3.2 Two flavors of rigid

- **Articulated rigid** (robot links). `body_q` / `body_qd` are authoritative. Joints, motors, limits, `body_parent_f` reporting from `SolverXPBD` work unchanged. A MorphIt-generated lattice (volume-packed by default) exists in the particle world solely so the link can participate in particle-based contact; the lattice does *not* carry the body's dynamics.
- **Free shape-matched rigid** (the cup, the dropped cube). Particles are authoritative. Shape matching + momentum conservation (SRXPBD) keeps the group rigid.

### 3.3 Three contact pipelines (all reduced to particle level)

1. *Particle ↔ analytical shape* (ground plane, container walls, mesh, primitives). `solve_particle_shape_contacts` — XPBD kernel reused.
2. *Particle ↔ particle, cross-phase*. `solve_particle_particle_contacts` — extended kernel handling lattice / SM-rigid / soft / fluid mix on either side.
3. *Articulated link ↔ articulated link*. Reduces to (2) at the particle level: shell-particle ↔ shell-particle, with both sides routing Δx → body wrench. **No analytical shape-shape contact code path in UXPBD.**

The cross-substrate contact dispatch matrix is in §6.

### 3.4 Compatibility with Newton's existing API

- `Model.body_q`, `Model.body_qd`, `Model.joint_*`, `State.body_parent_f` — unchanged semantics for articulated rigid bodies.
- `Model.particle_q`, `particle_qd`, `particle_radius`, `particle_inv_mass`, `particle_flags` — unchanged; UXPBD adds metadata fields (§3.1) but populates them through existing builder calls plus new ones.
- `Contacts.force` — exposed for analytical-shape contacts and lattice-sphere contacts via the same array.
- `SolverNotifyFlags` (BODY_PROPERTIES, BODY_INERTIAL_PROPERTIES) — observed, refresh kinematic state as in `SolverXPBD`.

## 4 Per-iteration solver loop

`SolverUXPBD.step()` algorithm:

```text
step(state_in, state_out, control, contacts, dt):

  # ─── 1. PREDICT ─────────────────────────────────────────────────────────
  integrate_particles(state_in → state_out)        # x* = x + v·dt + g·dt² ; UPPFRTA mass-scaling for stacks
  integrate_bodies   (state_in → state_out)        # XPBD-style with control.joint_f, gravity, body_f
  update_lattice_world_positions(state_out.body_q → lattice particles in state_out.particle_q)
                                                   #   x_i = body_q[link(i)] ⊗ p_rest_i  (kinematic projection)
                                                   #   writes particle_radius from (lattice_r - lattice_delta);
                                                   #   in v1 lattice_delta == 0, so particle_radius == lattice_r

  # ─── 2. CSLC v2 HOOK ───────────────────────────────────────────────────
  self.compute_compliant_contact_response(...)     # v1: no-op. v2: solve K δ = k_c (φ^rest - δ)_+

  # ─── 3. NEIGHBORS & COLLISION DETECTION ────────────────────────────────
  model.particle_grid.build(state_out.particle_q, radius = particle_max_radius·2·expand_factor)
  model.collide(state_out)                          # populates Contacts.rigid_contact_* (analytical shape pairs)
                                                    # and Contacts.soft_contact_* (particle ↔ analytical shape).
                                                    # Particle ↔ particle contacts are NOT pre-listed here;
                                                    # they are discovered inline by solve_particle_particle_contacts
                                                    # via model.particle_grid queries during step 5b.

  # ─── 4. STABILIZATION SUB-LOOP (UPPFRTA §4.4, positions only, no KE injection)
  # Helpers `apply_position_only` / `apply_body_wrench_position_only` are variants of
  # apply_particle_deltas / apply_body_deltas that skip the velocity-update branch
  # (so the stabilization pass corrects positions without injecting kinetic energy).
  for s in range(stabilization_iterations):
      Δx ← 0 ;  body_wrench_iter ← 0
      solve_particle_shape_contacts        → Δx, body_wrench_iter
      solve_particle_particle_contacts     → Δx                            # phase-gated; lattice-lattice goes here
      apply_position_only(Δx)                                               # NO velocity update
      apply_body_wrench_position_only(body_wrench_iter)                     # for lattice-bound articulated bodies

  # ─── 5. MAIN CONSTRAINT LOOP ───────────────────────────────────────────
  for i in range(iterations):

      # ── 5a. Optional grid rebuild for stiff/fast scenes ──
      #   threshold = grid_expand_factor · model.particle_max_radius  (see §4.3)
      if grid_rebuild_policy == "every_iteration" or
         (grid_rebuild_policy == "adaptive" and max_displacement_since_last_build > threshold):
          model.particle_grid.build(state.particle_q, ...)
          model.collide(state)

      Δx ← 0 ;  body_wrench_iter ← 0

      # ── 5b. Particle-level constraints (all parallel into Δx) ──
      solve_particle_shape_contacts         → Δx, body_wrench_iter         # particle ↔ static shape
      solve_particle_particle_contacts      → Δx                            # cross-phase; UPPFRTA µ_s/µ_k friction inline
      solve_fluid_density_constraints       → Δx     [if fluid_phase_count > 0]   # PBF unilateral ρ/ρ₀ - 1 ≤ 0
      solve_springs                         → Δx     [if model.spring_count > 0]
      solve_bending                         → Δx     [if model.edge_count > 0]
      solve_tetrahedra                      → Δx     [if model.tet_count > 0]
      apply_particle_deltas(Δx, dt)                                         # divide by n_i count, SOR ω; update x, v

      # ── 5c. Free shape-matched rigid groups (SRXPBD, momentum-correct) ──
      if num_dynamic_shape_matched_groups > 0:
          solve_shape_matching_batch_tiled  → Δx, store p_linear, p_angular per group
          apply_particle_deltas(Δx, dt)
          enforce_momentum_conservation_tiled                                # SRXPBD: subtracts non-conservative comp.

      # ── 5d. Articulated rigid (wrench accumulation → body, then joints) ──
      if articulated_body_count > 0:
          accumulate_lattice_wrench_into_body_deltas(body_wrench_iter)
          apply_body_deltas(body_q, body_qd, body_deltas, dt)
          solve_body_joints                  → body_deltas, joint_impulse   # XPBD joint loop, motors, limits
          apply_body_deltas(body_q, body_qd, body_deltas, dt)
          update_lattice_world_positions(body_q → lattice particles)        # re-sync lattice to new body_q

  # ─── 6. RESTITUTION  (optional, post-loop, XPBD-style) ─────────────────
  if enable_restitution:
      apply_particle_shape_restitution
      apply_particle_particle_restitution                                   # new: cross-body soft-soft etc.
      apply_rigid_restitution                                               # articulated only

  # ─── 7. WRENCH & FORCE REPORTING ───────────────────────────────────────
  convert_joint_impulse_to_parent_f → state_out.body_parent_f               # XPBD existing
  if contacts.force is not None:
      convert_contact_impulse_to_force                                      # analytical-shape contacts
      convert_lattice_contact_impulse_to_force                              # new: lattice-sphere contact wrench

  copy_kinematic_body_state(state_in → state_out)
```

### 4.1 Five novel kernels vs. the three source solvers

1. **`update_lattice_world_positions`** — projects `body_q` onto the lattice particles. Writes both `particle_q` (= `body_q ⊗ p_rest`) and `particle_radius` (= `lattice_r - lattice_delta`; in v1, `lattice_delta == 0` always). The `particle_radius` write is the load-bearing CSLC seam: in v2, CSLC writes nonzero `lattice_delta` and downstream contact kernels automatically see a compressed effective radius.
2. **`accumulate_lattice_wrench_into_body_deltas`** — for every lattice-sphere contact that produced Δx_i in step 5b, compute the body-equivalent spatial wrench `(Σ Δx_i, Σ r_i × Δx_i)` using the body's true `body_inv_mass` / `body_inv_inertia`, accumulate per body. Treats the lattice as a force probe, not a mass-bearing element. Implements the Hard-Kinematic Shell choice (Q5-A).
3. **`solve_fluid_density_constraints`** — PBF density kernel (Macklin & Müller 2013), unilateral `ρ_i/ρ_0 - 1 ≤ 0`, gated to fluid phases. Solid particles contribute to fluid density per UPPFRTA §7.1 eq. 27 with scaling factor `s`.
4. **`apply_particle_particle_restitution`** — restitution between cross-phase particle contacts where at least one side is a soft body. Skipped between two shape-matched rigid groups (those use rigid restitution via shape-match recovery).
5. **`convert_lattice_contact_impulse_to_force`** — populates `contacts.force` for lattice-sphere contacts so paper-grade force traces work.

### 4.2 Iteration count parameters

- `stabilization_iterations` (default `1`): UPPFRTA §4.4 positions-only pre-pass. Fixes initial penetration without injecting kinetic energy. Increase to `2–3` for scenes with severe initial penetration (e.g., gripper closing on a rigid object).
- `iterations` (default `4`): main constraint loop. Compares to `SolverXPBD`'s default `2` — increased because UXPBD also iterates fluid density and shape matching, which converge slower.

### 4.3 Grid rebuild policy (made explicit per reference audit)

```python
grid_rebuild_policy: Literal["once_per_step", "every_iteration", "adaptive"] = "adaptive"
```

- `"adaptive"` (default): grid built once after step 3 with expanded search radius `particle_max_radius · 2 · expand_factor` (UPPFRTA recommendation). Rebuilt inside the main loop only if any particle moved more than `expand_factor · particle_max_radius` since the last build.
- `"every_iteration"`: rebuild at the top of every main-loop iteration. Matches the reference CUDA implementation behavior. Slowest but safest for stiff/fast scenes.
- `"once_per_step"`: build once, never rebuild. Documented as unsafe with large `iterations` count or fast-moving bodies. Use only when measured displacement < `expand_factor · particle_max_radius`.

### 4.4 Final velocity update timing

UXPBD's `apply_particle_deltas` writes both `particle_q` and `particle_qd` per iteration using `qd = (q - q_init) / dt` where `q_init = state_in.particle_q` (initial position at the start of the step). Because `q_init` is invariant across iterations, the last iteration's output is mathematically equivalent to UPPFRTA's once-after-loop `v = (x* - x) / dt`. No spurious kinetic energy is injected. The XPBD per-iteration pattern is retained for code-path consistency with `SolverXPBD`.

## 5 Lattice (MorphIt-generated, variable-radius) + CSLC v2 seams

### 5.1 Data model

Per-lattice-sphere data stored in flat CSR-indexed arrays:

```python
# Per-link CSR offsets
model.link_lattice_sphere_start: wp.array[wp.int32]
model.link_lattice_sphere_count: wp.array[wp.int32]

# Per-lattice-sphere state (concatenated across all links)
model.lattice_p_rest:       wp.array[wp.vec3]    # rest position in link body frame (MorphIt)
model.lattice_r:            wp.array[float]      # rest radius (MorphIt, non-uniform)
model.lattice_normal:       wp.array[wp.vec3]    # outward normal in body frame (surface spheres only)
model.lattice_is_surface:   wp.array[wp.uint8]   # 1 for surface spheres, 0 for interior
model.lattice_link:         wp.array[wp.int32]   # host link index
model.lattice_particle_index: wp.array[wp.int32] # slot in state.particle_q

# ── CSLC v2 seams (present in v1, unused) ──
model.lattice_delta:        wp.array[float]      # scalar compression δᵢ; v1: zero. v2: CSLC writes.
model.lattice_delta_prev:   wp.array[float]      # warm-start buffer for v2 damped Jacobi
model.lattice_K_diag:       wp.array[float]      # K diagonal entries k_a + k_ℓ |N(i)|
model.lattice_neighbors_csr:    wp.array[wp.int32]  # adjacency for lateral coupling
model.lattice_neighbors_offset: wp.array[wp.int32]
model.lattice_k_anchor:     wp.array[float]      # k_a per sphere; v1: stored, never read.
model.lattice_k_lateral:    wp.array[float]      # k_ℓ per sphere
model.lattice_k_contact:    wp.array[float]      # k_c per sphere (calibrated from k_e_bulk)
model.lattice_damping:      wp.array[float]      # d_c per sphere
```

Lattice spheres **are** particles in `state.particle_q`. Their metadata distinguishes them: `particle_substrate == 0`, with `particle_group` pointing to the host link.

### 5.2 Forward kinematic update (the `particle_radius` seam)

```python
@wp.kernel
def update_lattice_world_positions(
    body_q:                wp.array[wp.transform],
    body_qd:               wp.array[wp.spatial_vector],
    body_com:              wp.array[wp.vec3],
    lattice_link:          wp.array[wp.int32],
    lattice_p_rest:        wp.array[wp.vec3],
    lattice_delta:         wp.array[float],         # v1: zero. v2: shrinks effective radius per CSLC eq. (1).
    lattice_r:             wp.array[float],
    lattice_particle_index: wp.array[wp.int32],
    particle_q:            wp.array[wp.vec3],
    particle_qd:           wp.array[wp.vec3],
    particle_radius:       wp.array[float],
):
    sid = wp.tid()
    link = lattice_link[sid]
    p_local = lattice_p_rest[sid]
    tf = body_q[link]
    pidx = lattice_particle_index[sid]

    particle_q[pidx]   = wp.transform_point(tf, p_local)
    r_world            = wp.quat_rotate(wp.transform_get_rotation(tf), p_local - body_com[link])
    omega              = wp.spatial_top(body_qd[link])
    v_lin              = wp.spatial_bottom(body_qd[link])
    particle_qd[pidx]  = v_lin + wp.cross(omega, r_world)
    particle_radius[pidx] = lattice_r[sid] - lattice_delta[sid]   # ← v1 = lattice_r ; v2 = lattice_r - δ
```

Called twice per `step()`: once after `integrate_bodies` (so contact detection sees post-prediction lattice), once at the end of every main-loop iteration (so subsequent iterations see lattice consistent with current `body_q`).

### 5.3 Effective inverse mass at a lattice sphere

For any contact involving a lattice sphere at world offset `r = x_i − p_body`, with contact normal `n`:

```
w_eff(i, n) = w_body[link] + (r × n)ᵀ · W_body_world[link] · (r × n)
```

Identical to `SolverXPBD`'s `solve_body_contact_positions` per-body effective inverse mass. Extracted into a device function `compute_lattice_sphere_w_eff` and called from every contact kernel that can touch a lattice sphere.

### 5.4 Backward pass — Δx on lattice → body delta

When a contact kernel computes Δx_i for a lattice sphere, instead of writing to `particle_q[i]`:

```python
if particle_substrate[i] == LATTICE:
    link = particle_group[i]
    r_world = particle_q[i] - body_com_world[link]
    wp.atomic_add(body_deltas[link],
                  wp.spatial_vector(
                      wp.cross(r_world, Δx_i),   # angular
                      Δx_i,                       # linear
                  ))
else:
    wp.atomic_add(particle_deltas[i], Δx_i)
```

`apply_body_deltas` (XPBD existing) then consumes `body_deltas` using the body's true inertia tensor.

### 5.5 CSLC v2 hook

```python
def step(self, ...):
    ...
    self.update_lattice_world_positions(state_out.body_q, state_out.body_qd)
    self.compute_compliant_contact_response(state_in, state_out, contacts, dt)   # ← v2 hook
    ...
```

- v1 default implementation: empty (no-op).
- v2 implementation: solve `K δ = k_c (φ^rest − δ)_+` via damped Jacobi (warm-started from `lattice_delta_prev`); update `lattice_delta` in place; re-run `update_lattice_world_positions` so `particle_radius` reflects new δ; optionally inject CSLC wrench directly into `state_in.body_f` and skip lattice-sphere contact deltas in step 5b (force-based CSLC path).

The hook signature is fixed in v1. Enabling `SolverUXPBD(model, enable_cslc=True)` in v1 raises `NotImplementedError` with a clear message pointing to the v2 milestone.

### 5.6 Volume vs surface packing

MorphIt packings can be volumetric or surface-only. Default: **volume-packed** for both articulated links and free shape-matched rigid bodies (matches existing `Box.add_morphit_spheres` behavior). Each sphere carries `lattice_is_surface` indicating whether it sits on the body boundary; v2 CSLC operates only on the surface subset. Interior spheres are pure collision probes in both versions.

## 6 Cross-substrate contact dispatch

Phase ID assignment defaults:

| Entity | Phase ID assignment |
|---|---|
| Articulated robot link | One phase per *link* (so finger ≠ forearm — finger can contact forearm) |
| Free shape-matched rigid body | One phase per *body* |
| Soft body | One phase per *body*; self-collision off by default |
| Cloth piece | One phase per *piece*; self-collision automatic through particle pipeline |
| Fluid (single phase in v1) | One phase per *fluid type* |
| Static analytical shape | n/a; particles collide with shapes regardless of shape's hosted body's phase |

Default rule: same `phase` → no inter-particle contact constraint. Cross-phase pairs go through `solve_particle_particle_contacts`. Fluid density (substrate 3) is the only intra-phase exception (fluid particles of the same phase contribute to each other's density but don't contact-collide).

**Contact pair dispatch matrix** (one kernel per cell; "mirror" = symmetric of the upper-triangle entry):

| ↓ p1 \ p2 → | Lattice | SM-rigid | Soft | Fluid | Static shape |
|---|---|---|---|---|---|
| **Lattice** | same link skip; diff link → `solve_particle_particle_contacts`; both route Δx → body wrench | `solve_particle_particle_contacts`; lattice → wrench, SM-rigid → Δx (SRXPBD re-enforces rigidity) | `solve_particle_particle_contacts`; lattice → wrench, soft → Δx | `solve_particle_particle_contacts` + fluid-solid coupling; lattice → wrench, fluid → Δx | `solve_particle_shape_contacts`; lattice → wrench |
| **SM-rigid** | mirror | same group skip; diff group → `solve_particle_particle_contacts`; momentum re-conserved | `solve_particle_particle_contacts` | `solve_particle_particle_contacts` + fluid-solid coupling | `solve_particle_shape_contacts` |
| **Soft** | mirror | mirror | v1: same body skip (self-collision off); diff body → `solve_particle_particle_contacts` | `solve_particle_particle_contacts` + fluid-solid coupling | `solve_particle_shape_contacts` |
| **Fluid** | mirror | mirror | mirror | same fluid phase → density only; diff phase → density + contact (v3+) | `solve_particle_shape_contacts` |
| **Static** | mirror | mirror | mirror | mirror | n/a |

### 6.1 Single unified `solve_particle_particle_contacts`

The extended kernel reads `particle_substrate[i]`, `particle_substrate[j]`, `particle_phase[i]`, `particle_phase[j]` and `particle_group[i]`, `particle_group[j]` per pair. Behavior per pair:

1. If `particle_phase[i] == particle_phase[j]` and not (fluid same-phase): skip (self-collision within body).
2. Compute Δx per UPPFRTA §6.1 contact + position-level Coulomb friction with `µ_s`, `µ_k`.
3. For each side's Δx routing: if substrate is `LATTICE`, accumulate spatial wrench on the host link's `body_deltas`; else accumulate `particle_deltas[idx]`.
4. Effective inverse mass: for lattice sphere use `compute_lattice_sphere_w_eff`; else use `model.particle_inv_mass[i]`.

This is the *only* contact kernel needed to cover all five cross-substrate mixed cases.

### 6.2 Fluid–solid coupling (UPPFRTA §7.1)

```
ρᵢ = Σ_{j∈fluid} W(xᵢ − xⱼ, h)  +  s · Σ_{j∈solid} W(xᵢ − xⱼ, h)
```

For UXPBD, every non-fluid substrate counts as "solid" in the density sum. Scaling factor `s` is exposed per substrate via `model.fluid_solid_coupling_s: wp.array[float]` (length 4, indexed by substrate).

### 6.3 Phase collision matrix (v2 lever)

```python
model.phase_collision_matrix: wp.array2d[bool]   # shape [phase_count, phase_count]
```

v1 default: all `True` except on the diagonal (`True` = collide). v2+ usable via `builder.set_phase_collision(a, b, enabled=False)` to selectively disable e.g. robot-self-collision optimization.


## 8 Test plan and phasing

### 8.1 Phase 1 — Articulated rigid + lattice + static contact

**Lands:** `SolverUXPBD` skeleton, `update_lattice_world_positions`, lattice data model, `add_lattice` / `add_lattice_to_all_links`, lattice-aware `solve_particle_shape_contacts`, XPBD joint + integration + restitution machinery (called, not copied), `compute_compliant_contact_response` no-op stub.

**Gate tests** (`uv run --extra dev -m newton.tests -k SolverUXPBD`):
- Every `SolverXPBD` joint test passes verbatim under `SolverUXPBD`.
- Pendulum period within 1% of analytical.
- Free-fall body trajectory matches `g·t²/2` within 0.5%.
- Lattice-spheres ↔ ground: link rests on plane, no oscillation > 1 mm at t = 2 s with `iterations=4`, `stabilization_iterations=1`.
- Lattice particle world positions match analytical FK at every step (single-link 6-DOF).
- Single lattice sphere of known mass dropped on ground: rest contact force == `m · g` within 2%.
- Static torque test: external torque on a link → `body_parent_f` matches XPBD output within 5%.

**Demo:** `example_uxpbd_lattice_push.py` — single revolute link with MorphIt-packed lattice rotates and pushes static cube. `test_final` asserts no inter-penetration at any frame.

**Exit criterion:** all gate tests green; ≥ 30 s headless demo runs without NaN/Inf.

### 8.2 Phase 2 — Free shape-matched rigid + UPPFRTA shock propagation

**Lands:** `solve_shape_matching_batch_tiled` + `enforce_momentum_conservation_tiled` (called from SRXPBD module), `particle_substrate` tagging, extended `solve_particle_particle_contacts` (all five mixed-substrate cases), UPPFRTA mass-scaling shock propagation, position-level Coulomb friction inline.

**Gate tests:**
- Existing `stack_two_cubes.py` analog passes under `SolverUXPBD`.
- **All seven analytical-solution tests from the PBD-R benchmark** pass on UXPBD shape-matched rigid bodies (free fall, free spin, projectile, restitution, friction, stacking, momentum exchange) — same tolerances as the PBD-R paper.
- Sand-pile angle of repose: 100 free shape-matched grains poured into a pile, measured angle matches `arctan(µ_k)` within 5°.
- Stack of 4 free shape-matched cubes settles to equilibrium with `shock_propagation_k=2.0` in < 500 ms simulation time.

**Demo:** `example_uxpbd_pick_and_place.py` — Scenario A. Articulated arm with lattice gripper grasps a MorphIt-packed cube, lifts, translates, releases on a stack. Tests: grasp distance < 2 × particle_radius at frame T_grasp; placed cube z-position matches target ± 1 cm at T_release.

**MuJoCo / Hydroelastic comparison (rigid-only):**
- Box-push: lattice gripper pushes free SM-rigid box across friction surface 0.5 m. End position match within 5%; trajectory L2 error vs. MuJoCo PGS < 10%; peak normal force within 20%. Reuses `cslc_mujoco/` harness.
- Ball-drop bounce: drop free SM-rigid ball from 1 m. UXPBD restitution disabled in both → both should converge to no-bounce within 5%.

**Exit criterion:** scenario A demo runs; MuJoCo box-push matches within tolerance; SRXPBD test suite passes.

### 8.3 Phase 3 — Soft bodies

**Lands:** `solve_springs`, `bending_constraint`, `solve_tetrahedra` integrated (called from XPBD module), soft body builders tagged for substrate=2, `set_self_collision` API; default off.

**Gate tests:**
- Hanging spring oscillation period within 2% analytical.
- FEM tet uniaxial stretch: force ↔ strain matches linear elasticity within 5%.
- Cloth drape over rigid sphere: max penetration < `particle_radius` at equilibrium.
- Soft cube compression by lattice gripper: cube returns to rest shape after release within 5% residual deformation.

**Demo:** `example_uxpbd_compliant_grasp.py` — Scenario B. Lattice gripper compresses soft FEM cube, lifts, releases. Test: grasp force trace shows compliant ramp (no impulsive spike); no FEM inversion; final shape within 5% of rest.

**Exit criterion:** scenario B demo runs without inversion; soft physics tests pass.

### 8.4 Phase 4 — Liquid

**Lands:** `solve_fluid_density_constraints` (PBF, single phase), `add_fluid_grid` / `add_fluid_particles` builders, fluid-solid coupling in density estimation, XSPH viscosity, optional cohesion.

**Gate tests:**
- Hydrostatic column: bottom-layer pressure ≈ `ρ · g · h` within 10%.
- Pouring mass conservation: fluid particle count source + receiver + airborne == initial at every frame.
- Dam-break flat plate: leading-edge position vs. analytical Ritter solution at t = 0.3 s within 15% (PBF approximate).
- Container holds water: cup filled with fluid, held stationary 5 s; loss < 1%.
- Buoyant rigid: free SM-rigid sphere density 500 kg/m³ floats ~50% submerged in 1000 kg/m³ fluid.

**Demo:** `example_uxpbd_pour.py` — Scenario D. Articulated gripper grasps cup → lifts → tilts → fluid pours into static bowl (`add_shape_mesh`). Test: post-pour bowl-fluid-count > 80% of initial cup fluid count.

**Exit criterion:** scenario D demo runs without NaN; mass conserved; hydrostatic within tolerance.

### 8.5 Phase 5 — Restitution polish, performance, optional gradient path

**Lands:** `apply_particle_particle_restitution` for soft-soft cross-body; `asv` benchmark suite per AGENTS.md; `requires_grad=True` smoke test (no crash).

**Gate tests:**
- Ball-drop restitution sweep `e ∈ {0.0, 0.3, 0.7, 1.0}`: rebound heights within 10% analytical.
- `asv` per-phase demo benchmarks within target ms/frame envelope.
- `requires_grad=True` smoke test on each demo: no runtime errors. Gradient correctness out of scope.

**MuJoCo comparison:** restitution sweep on rigid ball drop vs. MuJoCo `solref` / `solimp`.

**Exit criterion:** v1 release candidate.

## 9 Implementation strategy and risks

### 9.1 Class hierarchy

```
SolverBase  (existing)
   └── SolverUXPBD  (new — does NOT inherit from SolverXPBD or SolverSRXPBD)
```

UXPBD calls into existing kernels in `newton/_src/solvers/xpbd/kernels.py` and `newton/_src/solvers/srxpbd/kernels.py` directly. No code duplication. Cross-solver kernels stay where they live; UXPBD imports them.

### 9.2 File layout

```
newton/_src/solvers/uxpbd/
    __init__.py
    solver_uxpbd.py                  # SolverUXPBD class
    kernels.py                       # new UXPBD-specific kernels
    lattice.py                       # lattice generation utilities (calls MorphIt loader)
    fluid.py                         # PBF density kernels + builder helpers
    coupling.py                      # fluid-solid coupling utilities
```

Public re-export at `newton.solvers.SolverUXPBD` via `newton/solvers.py`.

### 9.3 Naming conventions (per AGENTS.md)

- Prefix-first: `add_lattice`, `add_lattice_to_all_links`, `add_fluid_grid`, `add_fluid_particles`.
- PEP 604 unions (`x | None`).
- `wp.array[wp.vec3]` bracket syntax.
- Google-style docstrings with SI units (`[m]`, `[kg]`, `[N·m]`).
- `:class:` / `:meth:` cross-refs to shortest public API path; never `newton._src`.

### 9.4 Risks

| Risk | Mitigation |
|---|---|
| **Stiff lattice-vs-ground stacks oscillate** under UXPBD because UPPFRTA shock propagation is per-particle, not per-body | Increase `iterations`, document tuning; consider per-body height heuristic as a v2 follow-up if observed |
| **Cross-substrate momentum mismatch** at lattice ↔ SM-rigid contact (different momentum policies per side) | The hybrid Q6 policy already accepts this; SRXPBD's post-pass re-conserves SM-rigid side; document the asymmetry |
| **MuJoCo comparison fails on box-push** due to friction model differences (regularized Coulomb in UXPBD vs. complementarity Coulomb in MuJoCo) | Use loose tolerance (10% trajectory L2); document as inherent to the solver class difference |
| **Variable particle radii break neighbor-search** for very wide radius distributions (small + large spheres) | Cap radius ratio at 1:10 (UPPFRTA recommendation); document as a MorphIt configuration requirement |
| **CSLC v2 wants force-based wrench but v1 architecture is position-based** | The seam supports both paths: v2 can either inject `body_f` and skip v1's particle-Δx routing, or run the CSLC quasistatic before contact and let v1's pipeline still operate on the resulting `lattice_delta`-compressed radii |
| **Fluid-rigid coupling unstable at very low fluid:rigid radius ratios** | Tune `fluid_solid_coupling_scale` per-substrate; expose to user; document the recommended ratio bound |

## 10 Out of scope (re-affirmed)

- CSLC quasistatic solve (v2)
- Soft body self-collision enabled by default (v3+)
- Multi-phase fluids, immiscible liquids (v3+)
- Gas / smoke (no demand)
- Gradient correctness validation (deferred)
- Equality / mimic joint constraints
- Hierarchical multi-resolution particles
- Runtime particle addition after `finalize()` (v3 if needed)
